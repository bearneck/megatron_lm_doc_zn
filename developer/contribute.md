# 为 Megatron-LM 做贡献

本文档概述了非英伟达贡献者向 Megatron-LM GitHub 仓库提交问题和拉取请求的流程与政策。

欢迎所有人参与项目贡献！我们最近已从使用内部仓库迁移到直接在 GitHub 仓库进行所有开发。

在贡献时，确保变更符合项目方向非常重要。修复错误的小改动是受欢迎和值得赞赏的。如果提议进行大的架构变更或出于风格原因的改动，请先创建一个议题，以便我们进行讨论。

## 议题政策

请务必提交您发现的任何错误，并注意以下几点：

-   如果提交错误报告，即您发现某些功能未按预期工作，请使用 BUG 模板。
-   如果您发现速度或准确度出现回归，请使用 REGRESSION 模板。
-   如果您请求新功能或修改现有功能，请使用 ENHANCEMENT 模板。
-   如果创建议题是为了提问，则不需要模板，但请使您的问题尽可能清晰简洁。
-   一个错误对应一个议题。将多个问题放在同一个议题中会使讨论和解决变得不必要的复杂。
-   如果我们能轻松复现您的错误，开发团队最有可能快速关注到它。
-   使用正确的拼写、语法和标点符号。
-   以权威和技术性的语气撰写。

## 代码提交政策

### 应该做的

-   以与被更改文件一致的风格格式化新代码。Megatron-LM（目前）还没有风格指南或强制执行的格式化规则。
-   将您的更改拆分为独立的、原子性的提交，即每个功能或修复一个提交。
-   确保您的提交是基于主分支（master）进行变基的。
-   使用祈使语气撰写提交信息的主题行（例如"Change the default argument for X"，而不是"Changed the default argument for X"）。
-   使用正确的英语、注意措辞和标点符号来撰写您的提交信息。
-   检查您的代码、注释和提交信息的拼写。

### 不应该做的

-   提交与项目许可证不兼容的代码。
-   触及 PR 所述范围之外的任何内容。这包括对与 PR 无关的代码进行格式化更改。
-   在多个提交中过度迭代您的设计。
-   包含被注释掉的代码。
-   在没有先创建议题讨论的情况下尝试进行大的架构变更。

## 议题和拉取请求问答

### 我已经提交了一个议题和 PR。我什么时候能得到一些反馈？

您应该在 2 个工作日内收到回复。
### 我需要帮助，应该联系谁？

请使用 [@mcore-oncall](https://github.com/orgs/NVIDIA/teams/mcore-oncall)。

### 如果我的问题或拉取请求没有得到关注，该怎么办？

在 2 个工作日之后，请标记用户 [@mcore-oncall](https://github.com/orgs/NVIDIA/teams/mcore-oncall)。

### 对于 X 天内未被处理的问题和拉取请求，是否有相关政策？它们应该被关闭吗？

是的，我们有一个机器人，会在 60 天后将未被处理的拉取请求标记为“陈旧”。

我们积压了大量可以追溯到几年前的问题和拉取请求。我们目前正尝试通过从后往前的方式对它们进行分类处理。对于我们认为可能仍然相关的旧问题，可能会收到要求使用最新代码重新测试的请求。如果没有回应，它们可能会被关闭。同样，如果您认为应该重新打开它们，只需回复一条评论说明情况即可。

谢谢！
